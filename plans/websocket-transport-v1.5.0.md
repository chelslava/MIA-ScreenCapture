# Техдизайн WebSocket transport для `v1.5.0`

> Обновлено: 2026-03-28
> Статус: проектирование перед внедрением

## Цель

Добавить real-time транспорт для доменных событий записи без изменения
доменных сервисов и бизнес-логики.

`v1.5.0` должен дать:

1. Подписку GUI и внешних клиентов на поток событий в реальном времени.
2. Единый контракт для событий, статуса и ошибок.
3. Совместимость с текущим `EventBus` и `WebSocketManager`.
4. Контролируемое подключение через аутентификацию и heartbeat.

## Текущий Контекст

В проекте уже есть:

1. `core/event_bus.py` с `RecordingEvent`, `RecordingEventType` и
   `InMemoryEventBus`.
2. `api/websocket.py` с `WebSocketManager`, который принимает события из
   event bus и хранит недавнюю историю.
3. `api/server.py`, который уже содержит API-слой и готов к расширению
   новым маршрутом WebSocket.

Ключевая идея: домен публикует события в `EventBus`, а transport-слой
подписывается на них и отвечает только за доставку, буферизацию и
протокол соединения.

## Архитектура

Планируется разделить решение на три слоя:

1. Доменный слой: `RecordingService` и другие компоненты публикуют
   `RecordingEvent`.
2. Агрегатор событий: `WebSocketManager` получает события из `EventBus`,
   нормализует payload и хранит историю.
3. WebSocket transport: отдельный сервер/эндпоинт отдает события
   подписчикам, управляет соединениями и состоянием клиентов.

Схема потока:

```text
RecordingService -> EventBus -> WebSocketManager -> WebSocket transport -> clients
```

## Источники Событий

Основные источники:

1. `RecordingEventType.STARTED`
2. `RecordingEventType.STOPPED`
3. `RecordingEventType.PAUSED`
4. `RecordingEventType.RESUMED`
5. `RecordingEventType.PROGRESS`
6. `RecordingEventType.ERROR`
7. `RecordingEventType.STATUS`

Дополнительные источники на следующем этапе:

1. Системные уведомления API.
2. Изменения статуса сервера.
3. Метрики transport-layer.

Важное правило: событие остается доменным объектом, transport не должен
знать о внутренней логике записи, кроме структуры payload.

## Формат Payload

Единый envelope для всех сообщений:

```json
{
  "type": "event",
  "channel": "recording",
  "event": {
    "type": "started",
    "timestamp": "2026-03-28T14:15:16.123456+00:00",
    "data": {
      "output_path": "D:\\\\Videos\\\\recording_20260328_141516.mp4"
    }
  },
  "meta": {
    "message_id": "01J...",
    "server_time": "2026-03-28T14:15:16.223456+00:00",
    "schema_version": 1
  }
}
```

Рекомендуемые поля:

1. `type` - тип сообщения транспорта.
2. `channel` - логический канал доставки.
3. `event` - полезная нагрузка доменного события.
4. `meta.message_id` - уникальный идентификатор сообщения.
5. `meta.server_time` - время формирования ответа сервером.
6. `meta.schema_version` - версия контракта payload.

## Каналы И Типы Сообщений

Предлагается ввести несколько каналов:

1. `system` - служебные сообщения о соединении, heartbeat, reconnect.
2. `recording` - события записи.
3. `api` - статус API сервера и ошибки transport-уровня.
4. `metrics` - сводные состояния и счетчики.

Типы сообщений:

1. `hello` - приветствие при подключении.
2. `snapshot` - начальное состояние клиента.
3. `event` - доменное событие.
4. `status` - актуальное состояние сервиса.
5. `heartbeat` - служебный ping/pong.
6. `error` - ошибка transport или запроса.
7. `ack` - подтверждение приема сообщения/команды.

Пример служебного сообщения:

```json
{
  "type": "hello",
  "channel": "system",
  "meta": {
    "schema_version": 1
  },
  "data": {
    "server": "MIA-ScreenCapture",
    "transport": "websocket",
    "protocol_version": "1.0"
  }
}
```

## Аутентификация

Аутентификация должна совпадать с текущим API-контуром:

1. Использовать `MIA_API_KEY` как приоритетный источник токена.
2. Поддержать передачу токена через заголовок `X-API-Key`.
3. Для WebSocket handshakes допускается query-параметр `token` только как
   fallback для ограниченных клиентов, но не как основной путь.

Рекомендуемый приоритет:

1. HTTP header.
2. WebSocket subprotocol или token parameter для совместимости.
3. Переменная окружения как источник истины на стороне сервера.

Правила безопасности:

1. Токен не логировать целиком.
2. Отдавать `401`/`4401` при неуспешной авторизации.
3. Закрывать соединение сразу после проверки токена, если он неверный.

## Reconnect И Backoff

Клиент должен уметь восстанавливаться после разрыва соединения.

Параметры по умолчанию:

1. `reconnect_initial_ms = 500`
2. `reconnect_max_ms = 15000`
3. `reconnect_multiplier = 1.8`
4. `reconnect_jitter = 0.2`

Поведение:

1. При штатном закрытии клиент не обязан переподключаться.
2. При сетевой ошибке клиент повторяет попытку с экспоненциальным
   backoff и jitter.
3. Сервер при переподключении отдает `snapshot` и последние события из
   буфера `WebSocketManager`.

## Heartbeat

Нужен двусторонний heartbeat, чтобы быстро замечать "мертвые" соединения.

Предложение:

1. Сервер отправляет `ping` каждые 15 секунд.
2. Клиент отвечает `pong` не позднее 5 секунд.
3. Если ответа нет в течение 3 интервалов, соединение закрывается.

Дополнительно:

1. В `meta` можно передавать `uptime_seconds`.
2. Если клиент долго не читает socket, сервер должен закрыть соединение
   вместо бесконечного буферизирования.

## Error Contract

Ошибка transport должна быть единообразной:

```json
{
  "type": "error",
  "channel": "api",
  "error": {
    "code": "unauthorized",
    "message": "Недействительный API токен",
    "details": [
      {
        "field": "token",
        "message": "Token is missing or invalid"
      }
    ]
  },
  "meta": {
    "schema_version": 1,
    "retryable": false
  }
}
```

Базовые коды ошибок:

1. `unauthorized`
2. `forbidden`
3. `validation_error`
4. `transport_unavailable`
5. `rate_limited`
6. `internal_error`

Правила:

1. Ошибка должна быть машиночитаемой.
2. Текст сообщения должен быть коротким и понятным пользователю.
3. Для повторяемых ошибок указывать `retryable: true`.

## Хранение Истории

`WebSocketManager` уже хранит недавние события в памяти.

Для `v1.5.0` предлагается:

1. Сохранить буфер последних `N` событий в памяти.
2. Отдавать историю при первом подключении или после reconnect.
3. Не делать persistent storage на первом этапе, чтобы не смешивать
   transport и долговременное хранение.

Параметр буфера:

1. `max_events = 500` как базовое значение.
2. Переопределение через конфиг допустимо на следующем этапе.

## Поэтапный План Внедрения

### Этап 1. Протокол И Контракт

1. Зафиксировать JSON schema для сообщений.
2. Описать каналы, коды ошибок и heartbeat.
3. Подготовить тесты на сериализацию и валидацию payload.

### Этап 2. Сервер И Подписки

1. Добавить WebSocket endpoint в `api/server.py`.
2. Подключить его к `WebSocketManager`.
3. Реализовать auth handshake и выдачу `snapshot`.
4. Протестировать reconnect и обработку разрыва.

### Этап 3. GUI И Клиент

1. Подключить GUI к real-time каналу.
2. Отображать события записи, статусы и ошибки.
3. Добавить визуальный индикатор соединения и heartbeat.

### Этап 4. Нагрузочная Проверка

1. Проверить несколько клиентов одновременно.
2. Проверить поведение при высокой частоте `PROGRESS`.
3. Проверить утечки памяти и рост очереди.

## Риски

1. Дублирование событий при reconnect.
2. Рост памяти из-за буфера истории.
3. Сложность синхронизации между GUI и API сервером.
4. Неустойчивость на Windows при слабом сетевом соединении.
5. Смешение transport и доменной логики.

## Метрики Готовности

Релиз можно считать готовым, если выполнены условия:

1. WebSocket соединение устанавливается и восстанавливается после
   искусственного обрыва.
2. `hello` и `snapshot` приходят при первом подключении.
3. `event` сообщения отображаются без потерь в типовых сценариях.
4. Heartbeat закрывает "мертвые" соединения.
5. Auth reject работает для неверного токена.
6. Unit-тесты покрывают сериализацию, reconnect и error contract.
7. Не растет RSS-память при длительной сессии со стабильным потоком.

## Открытые Вопросы

2. Должен ли `snapshot` содержать только последние события или еще
   агрегированное состояние записи.
3. Нужен ли rate limit на количество reconnect в минуту.
