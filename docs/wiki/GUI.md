# GUI

## Вкладки интерфейса

- `Запись`: старт, пауза, остановка, параметры записи.
- `Планировщик`: создание и редактирование задач записи.
- `Диагностика`: проверка окружения и состояния компонентов.
- `API`: управление API-сервером и просмотр API-логов.

## Вкладка API

Доступно во вкладке:

- Настройка порта API (`1..65535`).
- Настройка API токена.
- Кнопка `Копировать токен` в буфер обмена.
- Кнопки `Запустить`, `Остановить`, `Перезапустить`.
- Кнопка `Открыть папку логов API`.
- Окно логов API в реальном времени (автообновление).

Поведение обновления:

- автообновление логов и статуса API активируется при открытии вкладки;
- когда вкладка скрыта, фоновое перечитывание логов останавливается;
- для журнала API показываются состояния загрузки, пустого журнала и
  ошибки чтения.

## Поведение токена

- При запуске приложения токен берется из `MIA_API_KEY`, если переменная задана.
- Если `MIA_API_KEY` не задана, используется сохраненный токен из `config/config.json`.
- При изменении токена во вкладке API значение синхронизируется в окружение процесса.

Подробнее: [Конфигурация и переменные окружения](Configuration-and-Environment).

## Поведение порта API

- Фактический порт берется из настроек API.
- Если Swagger открыт на `5010`, проверяйте запросы тоже на `5010`.
- Запросы к `5000` будут работать только если сервер реально слушает `5000`.

## Recent recordings

- Просмотр списка последних файлов.
- Фильтрация по имени и дате.
- Открытие файла и папки.
- Очистка списка последних записей.

## Загрузка системных данных

- список окон захвата обновляется в фоне и показывает loading/empty/error
  состояние вместо синхронной блокировки интерфейса;
- список микрофонов обновляется в фоне и пытается сохранить предыдущий
  выбор пользователя после refresh;
- API-лог читается через фоновый worker, поэтому вкладка `API` больше
  не перечитывает файл синхронно в UI-потоке;
- ручные кнопки `Обновить` для окон, микрофонов и API-логов сохраняются.

## Отзывчивость GUI

- Таймер отображения времени записи не работает в idle-состоянии.
- При скрытии главного окна timer времени записи приостанавливается и
  возобновляется при повторном показе окна.
- Первичная проверка FFmpeg выполняется в фоне и не должна блокировать
  создание главного окна.

## Readiness перед стартом записи

- На вкладке `Запись` есть compact readiness center с inline checklist и
  быстрыми действиями для частых проблем.
- Перед запуском записи GUI собирает readiness snapshot и проверяет:
  - доступность FFmpeg;
  - доступность пути вывода и свободного места;
  - доступность выбранного окна при `window capture`;
  - доступность микрофона для режимов с входным аудио.
- Блокирующие проблемы останавливают старт до запуска runtime-записи.
- Warning-сценарии не блокируют старт, но показываются inline в readiness
  center и через status bar.
- При блокировке окно переключается к вкладке `Диагностика`, которая
  использует ту же health-модель и показывает единый checklist.
- Tray/hotkey start path в GUI использует тот же preflight, потому что
  делегирует запуск через `MainWindow.start_recording()`.
- Стартовая проверка FFmpeg не использует отдельный modal dialog:
  предупреждение приходит через readiness center и non-modal feedback.

## Горячие клавиши и навигация

Текущий keyboard-first слой в главном окне:

- `Ctrl+R` — начать запись
- `Ctrl+P` — пауза / продолжить
- `Ctrl+S` — остановить запись
- `Ctrl+Shift+O` — открыть последнюю запись
- `Ctrl+Alt+O` — открыть папку выбранной записи
- `Alt+1` — вкладка `Запись`
- `Alt+2` — вкладка `Планировщик`
- `Alt+3` — вкладка `Диагностика`
- `Alt+4` — вкладка `API`
- `Ctrl+Alt+L` — открыть папку логов приложения

Дополнительно:

- для ключевых кнопок и status widgets добавлены базовые
  `accessibleName` / `accessibleDescription`;
- в главном окне задан упрощённый tab order для основных recording
  controls и списка последних записей.
- переходы по вкладкам и открытие логов работают через единый desktop
  action registry и доступны как из кнопок, так и через shortcuts.

Дополнительное покрытие accessibility:

- secondary views (`API`, `Захват`, `Аудио`, `Видео`, `Вывод`,
  `Диагностика`) тоже маркируют ключевые controls и status labels через
  `accessibleName` / `accessibleDescription`;
- tooltip-подсказки для ключевых действий синхронизированы с action
  descriptions.

## Non-modal feedback

- recoverable ошибки в start/API flow больше не завязаны только на modal
  dialogs: они дублируются через `status bar` и стандартный error signal;
- readiness-блокировки, ошибки валидации области захвата и типовые API
  operation failures доступны пользователю без обязательного modal окна.

## Единая тема и стили

- common title/status styles постепенно переводятся на
  `gui.styles.theme.Theme`;
- theme layer уже используется для ключевых status labels в `MainWindow`,
  `API`, `Захват`, `Аудио` и `Диагностика`;
- это подготавливает дальнейший вынос остальных inline `setStyleSheet()`
  в единый theme/style слой.

## Состояния кнопок записи

- idle / recording / paused / stopping состояния в `MainWindow` теперь
  применяются через единый helper `_update_ui_state(...)`;
- это снижает риск рассинхронизации между start/stop/pause flow и
  упрощает дальнейшие UX-изменения вокруг записи.
